# PRD — Clinic OS V1

> Last updated: 2026-05-02
> Author: Chloe + 阿寶

---

## 1. 產品定位

一套給小型診所使用的雲端營運系統。整合：

1. 病人資料
2. 病歷紀錄
3. 開藥與處方
4. 藥物庫存（含批號、效期、FEFO）
5. 收據與費用
6. 病假紙、轉診證明
7. AI 草稿、提醒、查漏
8. Audit log
9. A2A-ready 外部 agent 接口

**不是 AI chatbot。是一套可靠的診所後端系統 + 受控的 AI assistant layer。**

---

## 2. 目標用戶

### 2.1 主要客戶
- 1–5 醫生規模的診所
- 老闆（owner）通常是其中一位醫生
- 總員工 2–8 人

### 2.2 用戶角色
| 角色 | 主要工作 | 關鍵限制 |
|---|---|---|
| Owner | 經營決策、用戶管理、報表、prompt 設定 | 唯一可作廢收據、改權限的角色 |
| Doctor | 看診、寫病歷、開藥、開文件 | 不能改價格主檔（除非是 owner） |
| Nurse | 協助發藥、量測、登記 | 不可做醫療決策 |
| Reception | 掛號、收費、印收據 | 預設不可看完整病歷 |

### 2.3 多診所支援
- 一個 user 可屬於多間 clinic
- 登入後選擇 active clinic
- 所有 API 都要檢查 `clinic_id` 一致性

## 3. 核心使用情境（User Stories）

### US-01：完整看診流程
> 作為前台，我要能搜尋病人、建立就診；作為醫生，我要能寫病歷、開藥、看到即時庫存與藥費；作為前台，我要能確認收費並印收據。

### US-02：開藥即時計算
> 作為醫生，我輸入「一次 1 顆、一天 2 次、3 天」時，系統要立刻顯示總顆數 6 顆、藥費 30、目前庫存 120 顆夠不夠。

### US-03：FEFO 扣庫存
> 作為老闆，我要確保系統永遠先扣最快過期的批號，不要讓藥報廢。

### US-04：收據作廢
> 作為老闆，我發現一張收據開錯了，我要能作廢它，留下原因，且庫存自動回補。

### US-05：AI 病歷草稿
> 作為醫生，我口述病歷後，AI 要整理成 SOAP 格式的草稿，我修改後才確認寫入正式病歷。

### US-06：低庫存提醒
> 作為老闆，我要每天看到低庫存與快過期藥物清單。

### US-07：開病假紙
> 作為醫生，我要能在看診時直接生成病假紙 PDF，AI 給草稿，我確認後產生正式 PDF。

### US-08：審計追蹤
> 作為老闆，我要能查所有重要操作（誰、什麼時候、改了什麼）。

❌ Lab order 完整流程（只預留 schema）

## 5. 非功能需求

### 5.1 效能
- 一般 API 回應 P95 < 500ms
- 開藥即時計算 P95 < 200ms
- PDF 生成 P95 < 3s

### 5.2 可靠性
- 庫存扣減必須是 atomic transaction
- 收據編號不可重複（DB unique constraint）
- 即使 AI 服務掛掉，核心看診流程仍可運作

### 5.3 安全性
- 所有 API 強制 HTTPS
- Firebase Auth token 驗證
- Database 不對外開放，只接 Cloud Run service account
- 病人資料不送去未授權的外部模型
- 所有重要操作寫 audit log

### 5.4 合規
- 病歷至少保留 7 年（diff 區域可調）
- Soft delete only（不可硬刪）
- 收據作廢留 reason + 原始資料

## 6. AI 邊界（最重要）

詳見 `AI_BOUNDARY.md`。一句話：

> **AI 永遠寫進 `ai_drafts`，永遠不直接寫正式表。人類確認後才寫入。**

## 7. 成功標準（V1 上線判斷）

完成後，系統至少應該能：

- [ ] 老闆建立診所與使用者
- [ ] 醫生登入並選擇診所
- [ ] 前台建立病人資料
- [ ] 醫生建立就診紀錄
- [ ] 醫生開藥，系統自動計算總顆數、藥費、庫存
- [ ] 醫生完成就診後，系統生成 invoice draft
- [ ] 前台確認收費後正式發收據、扣庫存、寫 stock_movements
- [ ] 收據作廢留 reason、回補庫存、寫 audit log
- [ ] 醫生可生成病假紙 / 轉診信 PDF
- [ ] AI 可生成草稿，但不能直接寫正式資料
- [ ] Owner 可查看低庫存、快過期、收入摘要
- [ ] 所有重要操作都有 audit log

## 8. 風險與緩解

| 風險 | 緩解 |
|---|---|
| 醫生在開藥時誤觸「完成就診」導致庫存提早扣 | 「完成就診」只 confirm prescription，不扣庫存；扣庫存在「確認收費」一步 |
| AI 幻覺寫錯藥名 | AI 只能寫到 `ai_drafts`，醫生確認時要看到 diff |
| 收據作廢後庫存沒回補 | `stock_movements` 強制 atomic with invoice void |
| 多診所資料外洩 | 每個 query 都過 `clinic_permission` middleware |
| Cold start 慢影響開藥即時計算 | min_instances=1（生產） |
